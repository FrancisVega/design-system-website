These guidelines apply for developers and writers working with Bluemix UI and documentation. They include guidelines for conversational style, as well as guidelines for capitalization, punctuation, and wording in the user interface. The guidelines are based on conversational style, which is one that engages the reader. They are based also on IBM style, which ensures that information is translatable and usable.

Do's

Be simple and precise

When possible, use shorter, simple terms rather than formal, longer ones. Keep sentences concise, simple, friendly, and punchy. State only what the user needs to know.  Eliminate redundancy.  Keep sentences short (15 words or less, if possible). Text should be quick to read, incisive, and engaging.

Use Active Voice

Use active voice as much as possible.  Active voice focuses on the performer of the action and is often clearer, shorter, and more direct than passive voice.

Exceptions
Passive voice is acceptable when any of these conditions are true:

  • The system performs the action.
  • It is more appropriate to focus on the receiver of the action.
  • You want to avoid blaming the user for an error, such as in an error message.
  • The information is clearer in passive voice (eg.“The file was deleted.” instead of “You deleted the file.”)

Use Second Person

In most cases, use second person (“you”) to speak directly to the reader. In the user interface, an exception is in the case of possessive adjectives.

You can use first person (“my”) in headings or labels that are very specific to the user or customer data.  For example, “My Account” or “My Usage”. In explanatory text for the heading or label, switch to second person (“your”). For example, “Your usage is calculated from the 1st day of the month.”

Dont's

Please, Thank You, Congratulations

Do not use “please” in any instructions in documentation or UI. Terms of politeness are superfluous, convey the wrong tone for technical material, and are not regarded the same way in all cultures.

“Please” is allowed in UI only when the user is being inconvenienced.  For example, “Indexing might take a few minutes. Please wait.”

Exclamation Marks ( ! )

In general, in both UI and documentation, exclamation points are not recommended.  However, exclamation points can be used to further conversational style on a very limited basis. If used, no more than one exclamation point should be used in a context such as a single window or a single documentation topic. 

Exclamation points must be used only positively, and never negatively.  For example: “Your IBM Bluemix account is ready!”, but not “You reached your usage limit!”

Questions ( ? )

In general, in both UI and documentation, questions can be used to further conversational style, but should be limited. Make sure that headings that use questions are meaningful. In UI, questions can be used as a confirmation prompt in a modal.

For example:

  • As a heading in UI or topical information: “What is Bluemix?” 
  • As a confirmation prompt in UI: “Are you sure that you want to close your document without saving your changes?”

“May”

Use “can” to indicate ability, or use “might” to indicate possibility.
